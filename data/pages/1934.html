Introduction

This is the last part of a three part series covering our experiences running a coding dojo to improve Python and TDD skills in our team. You can read part 1 here and part 2 here.

Looking back

After the second dojo session, covered in part 2, we had concluded the following:

  1. The TDD process was not being following as we’d expected, despite the fact that the team really seemed to be giving it their best
  2. General code quality issues such as style guide (PEP8) compliance, error handling and the decoupling of code had been visibly improved
  3. We had successfully shown some of the team members how unpleasant it was to work with unclean code

In general therefore, we were pleased, but in this last session we really wanted to push the TDD process.

The Exercise

Pairing with one teacher per pair

We decided to split the team into pairs and then have one teacher with each team. Each pair would have one person writing the tests and one person getting the tests green, with the pairs swapping roles periodically. With each pair would be a teacher who would be able to see where things were falling down and then jump in quickly to help out.

More dogfooding…

Since the first round of dogfooding (covered in part 2) had been so successful we decided to rotate the code again. Each pair in the third session had to work on somebody else’s code from the second session.

The last new feature

Some of the team members were getting close to finishing all the previously specified features so we introduced one extra feature which we hoped would cause a lot of pain for people with highly coupled implementations:

  • The 2d grid is now 3d, but instead of wrapping like the other axes it is instead like a 2 layer car park. There is no wrapping in the z axis. e.g. The total grid is now two 2d grids, one above the other. This of course also means that there are now two new directions for the rover; up and down.

As with the last session, this new feature was to be completed after all previous features.

Ciaran Ó Néill

Our first kata series is a parable of success by way of failure. The goal of the kata series was to learn TDD, with an eye to clean(er) code. Thinking in terms of well known learning models (shu-ha-ri, Dreyfus, unconscious incompetence-conscious incompetence-etc) and assessing the team as novices in TDD we gave them detailed context-free instruction. The mistake we made was in assuming that the team wanted to learn what we had decided to teach them.

In inspecting & adapting after the first kata, Mark & David asked the guys to work on somebody else’s code in the second kata – this resulted in epiphanies all round. Getting the team to follow the TDD cycle in the third kata was straightforward now that they had viscerally understood why clean code was desirable. Another improvement in the third kata was the close guidance from the teachers. In conclusion – when teaching to novices, detailed context-free instruction is the guts of a successful approach but it is essential to also explicitly link the students general motivation to the projected outcome and to ensure the student has the level of guidance required, every step of the way.

The most important outcome to the kata series was that the team felt they had learned something valuable and reported that the production code that they wrote began to improve as of the first kata. Kudos to teachers & students!

Mark Coleman

write a failing test -> implement just enough code to make the test pass -> refactor the code -> repeat

Whilst the above seems very straight-forward, as I watched the pairs tackling the Mars Rover Kata I realised that for a beginner, every one of these ‘simple steps’ contains many tiny little questions to which the answers are not intuitively obvious.

Having a teacher present in the third session helped to move the teams through these roadblocks as they arose. This resulted in the team being able to better concentrate on the TDD process itself.

The most unexpected win however, came from exposing the team to someone else’s code. After struggling with unclean code, the team really wanted to learn how to avoid it, with one team member noting that “Readability is the most important!”

Wherever possible, I will be including code-swaps in future coding dojo sessions.
