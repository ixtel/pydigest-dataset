When we simply call locals() function it returns a dictionary same as when you call globals(), this is because the current scope is of the function main() which is where you are currently in.When you call locals() inside of a function it returns a dictionary of variables only inside that of the function scope. VARIABLE LIFETIME

The variable which lives inside the function cannot be accessed from the outside of the function. The variable lives only till the function is active. Afterwards it gets deleted. When a variable is created or modified its done in the current local scope if the variable is accessed python first searches inside the local scope first and then moves to the consecutive enclosing scope and finally to the global scope. This is the variable resolution rule used in python.

Lets see how we can modify a mutable global variable.

 

>>> x=[1,2,3,4]

 

>>> def foo():

 

        x[2]=0

 

        return x

 

>>> foo()

 

[1, 2, 0, 4]

 

>>> x

 

[1, 2, 0, 4]

 

When a global variable is used in python, then it can be modified if and only if its of mutable data type. Assigning simply creates a new variable and that will still be in local scope.

>>> def order(List):

 

        return sorted(List)

 

>>> order([1,4,3])

 

[1, 3, 4]

 

>>> def rev_order(List):

 

        return sorted(List,reverse=True)

 

>>> rev_order([1,4,3])

 

[4, 3, 1]

 

>>> def ordering(func,List):

 

        return func(List)

 

>>> ordering(rev_order,[54,23,12,90])

 

[90, 54, 23, 12]

 

>>> ordering(order,[54,23,12,90])

 

[12, 23, 54, 90]

 

We saw such functions which take up other functions as arguments in many built-in functions of python like sorted itself where we have a keyword argument called key which takes  up a function. 

CLOSURES

This is an important topic but rather than having a separate article I thought it would be easier by explaining it here. Lets first elaborate on how to nest a function inside a function inside a function(fooception)!

>>> def foo_out():

 

        i=10

 

        def foo_in():

 

                print(i**2)

 

        foo_in()

 

So just imagine what could happen when I call foo_out(). Remember the variable resolution rules stated before.

 

So python first started foo_out with i=10 then a foo_in() was created where i(first was looked in the scope of foo_in after not finding i it went to the enclosing scope of foo_out to find i) was squared and then at the end foo_in() was called. And hence we have 100.

What happens when we simply just return foo_in rather than calling foo_in?

>> def foo_out():

 

        i=10

 

        def foo_in():

 

                print(i**2)

 

        return foo_in

 

>>> foo_out()

 

<function foo_out.<locals>.foo_in at 0x1059ba158>

 

>>> foo_out()()

 

100

 

Seems straight forward doesn’t it?

But think about the resolution rules. So when we called foo_in() by calling foo_out()(), python checked for i in scope of foo_in, then it checks the scope it is in! Where is foo_in() being called from? Yes! main() so logically this should give an error that i doesn’t exist. The variable i should only be present when inside the scope of foo_out(). But we called foo_in() inside main(). That doesn’t seem right now! Every time we call foo_out we create and return foo_in. Somehow against the rules the function foo_in remembers the namespace where it was created in. This phenomenon is called as function closure. This in normal simple terms means that when a function is defined in any scope rather than global, it will keep a reference of the namespace where it was declared in.Tweaking the functions for better understanding:

>>> def foo_out():

 

        i=10

 

        def foo_in():

 

                print(i**2,locals())

 

        return foo_in

 

>>> foo_out()()

 

100 {'i': 10}
